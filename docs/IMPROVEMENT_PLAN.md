# AiTODO 改进计划（执行顺序版）

## 0. 执行说明
- 本计划严格按以下顺序执行：**阶段2 → 阶段4 → 阶段3 → 阶段1**。
- 每个阶段结束前必须完成：代码合并、文档更新、风险复盘。
- 执行过程中每小时更新一次进展文档（`docs/` 下新增或更新状态记录）。

---

## 阶段2：架构与数据升级（先做）

### 目标
- 降低业务耦合，提升可维护性与可扩展性。
- 完成本地数据层升级，为后续功能扩展提供基础。

### 工作项
1. 业务分层：将当前 Provider 内业务逻辑拆分为 `Repository + UseCase + Provider`。
2. 数据迁移：从 `SharedPreferences` 迁移到 `Isar`（含迁移脚本与回滚方案）。
3. 导入导出版本化：引入 `schemaVersion`，支持旧版本 JSON 自动兼容。
4. 优先级模型重构：支持“内置优先级 + 自定义优先级”统一排序与持久化。

### 交付物
- 新分层目录结构与迁移说明文档。
- 数据迁移代码、回滚开关、兼容测试样例。

### 验收标准
- 任务 CRUD、回收站、提醒、导入导出全链路在新数据层可用。
- 导入历史数据不丢字段（子任务、提醒、标签、完成状态）。

---

## 阶段4：产品能力增强（第二步）

### 目标
- 在新架构基础上快速提升用户可见价值。

### 工作项
1. AI 解析增强：规则引擎 + 模型接口双通道，失败自动降级到本地规则。
2. 任务视图增强：增加日历视图与热力图统计。
3. 提醒增强：稍后提醒、重复提醒、开机重建提醒。
4. 可用性增强：无障碍标签、字号适配、关键路径交互优化。

### 交付物
- 新页面与交互说明。
- AI 能力开关配置与降级策略文档。

### 验收标准
- 新功能不破坏既有任务流程。
- 弱网/离线场景下核心功能可用。

---

## 阶段3：测试与质量门禁（第三步）

### 目标
- 对前两个阶段改动建立稳定的质量防线。

### 工作项
1. 单元测试：覆盖 `TaskProvider/UseCase/Repository` 核心逻辑。
2. Widget 测试：覆盖任务列表、表单、导入导出、设置页关键流程。
3. Golden 测试：覆盖关键页面 UI 回归。
4. Lint 收紧：启用更严格规则（异步、空安全、动态调用）。

### 交付物
- 测试用例与覆盖率报告。
- 质量基线文档（失败即阻断合并）。

### 验收标准
- 核心模块测试覆盖率达到约定阈值（建议 >= 70%）。
- 新增功能必须附带测试。

---

## 阶段1：发布稳定化（最后收口）

### 目标
- 在功能与测试完成后，统一完成发布与运维闭环。

### 工作项
1. CI 流水线：`flutter analyze`、`flutter test`、`flutter build apk --release`。
2. 发布签名与环境：`debug/staging/release` 配置隔离。
3. 监控与告警：崩溃、导入失败、通知失败埋点与看板。
4. 文档收口：发布说明、回滚手册、已知问题清单。

### 交付物
- 可重复执行的发布流程。
- 发布包、变更日志、回滚方案。

### 验收标准
- 发布流程可由非开发成员按文档独立执行。
- 线上关键错误可在监控面板追踪定位。

---

## 里程碑建议
- M1（阶段2完成）：架构稳定，数据迁移完成。
- M2（阶段4完成）：核心新增能力上线可用。
- M3（阶段3完成）：质量门禁建立并生效。
- M4（阶段1完成）：具备稳定发布能力。
